## 1. Критика CVD-подхода — почему не работает

### Что мы делали
Брали ΔCVD (cumulative volume delta) на 5-минутных барах, z-score за 20 периодов, threshold > 0.6 → long/short.

### Почему это не может работать на MOEX

**Фундаментальная причина**: CVD — это **артефакт торговой архитектуры**, а не индикатор дисбаланса.

На MOEX:
1. **Нет реального агрегированного CVD** — vol_b (объём покупок) и vol_s (объём продаж) в tradestats_fo — это не CVD в классическом понимании (tick-level trade-by-trade). Это просто суммарные объёмы по Market Order Buy/Sell за бар, и они практически равны (корреляция ~0.99).
2. **Дисбаланс на MOEX микроскопический** — disb (disbalance) = |vol_b - vol_s| / (vol_b + vol_s). На MOEX disb < 0.05 в 95% баров. На стакане США disb может достигать 0.3-0.5.
3. **Латентность ≠ сигнал** — Разница vol_b - vol_s на 5-минутном баре — это просто шум округления, а не информационный поток. MM на MOEX дробит ордера, и разница объёмов на 5m — белый шум.
4. **Корреляция ΔCVD→return = -0.0029** — буквально ноль. Никакого edge нет и не может быть, потому что сам конструкт не несёт информации на MOEX.

**Что мы упустили**: CVD работает на тиковых данных с полной лентой (все сделки по возрастанию). На MOEX tradestats_fo — это **5-минутные агрегаты**, где информация о последовательности сделок (order flow) полностью потеряна. Это как пытаться определить направление ветра по средней температуре за день.

## 2. 10 идей новых сигналов

### Идея 1: Stop Hunt Detection (alerts_fo + prices_5m_oi)

**Источник данных**: `alerts_fo` (331K записей, 99.9% событий) + `prices_5m_oi` (5m бары)

**Сигнал**: В alerts_fo содержатся стоп-события. Когда цена пробивает локальный high/low (стоп-лоссы массово срабатывают) и затем возвращается обратно в диапазон — это классический stop hunt MM.

**Логика**:
1. Найти бары, где цена вышла за high/low последних N баров
2. Проверить, есть ли в alerts_fo записи о стоп-аукционах в этот момент
3. Если цена закрылась внутри диапазона после пробоя → разворот

**Формула сигнала**:
```
bar.high > max(high[-20:-1]) AND bar.close < bar.high - 0.3*(bar.high - bar.low) 
  → SHORT (false breakout upward)
bar.low < min(low[-20:-1]) AND bar.close > bar.low + 0.3*(bar.high - bar.low)
  → LONG (false breakout downward)
```

**Реализуемость**: 9/10 — данные есть, логика простая, тест за 1 день

### Идея 2: HHI + Price Divergence (hi2_fo)

**Источник данных**: `hi2_fo` (1.14M записей, HHI-концентрация) + `tradestats_fo` (цены)

**Сигнал**: HHI (Herfindahl-Hirschman Index) показывает концентрацию торгов. Когда HHI растёт (торговля концентрируется у MM/крупных участников) а цена падает → MM набирает лонг. Когда HHI растёт а цена растёт → MM раздаёт (распределение).

**Логика**:
1. z_HHI = z-score(HHI, 20 баров)
2. z_price = z-score(close, 20 баров) 
3. Если z_HHI > 1.5 и z_price < -1.0 → LONG (MM аккумулирует на падении)
4. Если z_HHI > 1.5 и z_price > 1.0 → SHORT (MM распределяет на росте)

**Реализуемость**: 8/10 — HHI данные есть, но нужно понять структуру hi2_fo (5m или 1d агрегация)

### Идея 3: FIZ/YUR Divergence на 5m (prices_5m_oi)

**Источник данных**: `prices_5m_oi` (64 символа, fiz_buy/sell + yur_buy/sell)

**Сигнал**: *Исправленная версия* — не просто fiz_net > 0, а divergence: fiz и yur идут в разные стороны с ускорением.

**Логика**:
1. fiz_flow_5m = (fiz_buy - fiz_sell) / total_oi  — нормализованный поток физиков за 5 минут
2. yur_flow_5m = (yur_buy - yur_sell) / total_oi
3. divergence = fiz_flow_5m + yur_flow_5m (знакопеременная): когда fiz и yur расходятся, yur обычно прав
4. Сигнал: divergence > 2σ И yur_flow в противоположную сторону от fiz_flow → торгуем по yur

**Ключевое улучшение против предыдущего теста**: 
- Не fiz_net как уровень, а **скорость изменения** fiz_net и yur_net
- divergence = diff(fiz_net) + diff(yur_net) — смотрим, куда уходит поток, а не где он стоит

**Реализуемость**: 7/10 — данные есть, но fiz/yur в prices_5m_oi только для 64 символов. Нужно проверить частоту обновления.

### Идея 4: Volume Profile S/R (tradestats_fo)

**Источник данных**: `tradestats_fo` — 5m бары, vol_sum = vol_b + vol_s

**Сигнал**: High Volume Nodes (HVN) — ценовые уровни с максимальным объёмом за сессию/неделю — работают как поддержка и сопротивление.

**Логика**:
1. На 5m барах за день построить volume profile: разбить ценовой диапазон дня на 20 зон, посчитать объём в каждой
2. Top-3 зоны по объёму = HVN (опорные уровни)
3. Когда цена подходит к HVN снизу → LONG (поддержка), сверху → SHORT (сопротивление)
4. Дополнительно: Point of Control (POC) — зона с макс. объёмом — как уровень открытия/закрытия

**Реализуемость**: 8/10 — данные есть, но построение volume profile на 5m требует агрегации. Можно сделать in-CH.

### Идея 5: Disb_z + OI_z Combo Breaker (tradestats_fo)

**Источник данных**: `tradestats_fo` (disb, oi_close) + `prices_5m_oi`

**Сигнал**: Комбинация z-дисбаланса и z-OI. Когда OI растёт (новые деньги входят) и дисбаланс максимальный → разворот.

**Логика**:
1. z_disb = z-score(|vol_b - vol_s| / (vol_b + vol_s), 40 баров)
2. z_oi = z-score(oi_close, 40 баров)
3. Сигнал LONG: z_oi > 1.0 (новые деньги) AND z_disb > 1.5 (экстремум дисбаланса) AND close < SMA(20)
4. Сигнал SHORT: z_oi > 1.0 AND z_disb > 1.5 AND close > SMA(20)

**Почему может работать**: OI growth + экстремальный дисбаланс = кульминация. Когда оба z-score одновременно высоки — это exhaustion, а не momentum.

**Реализуемость**: 7/10 — требует осторожности с look-ahead на z-score

### Идея 6: Session Pattern Exploitation (tradestats_fo + alerts_fo)

**Источник данных**: `tradestats_fo` + `alerts_fo` (время 10:00-23:50 MSK)

**Сигнал**: MOEX имеет 3 чёткие сессии:
- Утро: 10:00-14:00 (открытие, волатильность)
- День: 14:00-18:45 (основная торговля)
- Вечер: 19:00-23:50 (вечерняя сессия)

**Логика**:
1. **Open Drive (10:00-10:30)**: Первые 30 мин после открытия — price discovery. Если объём > 2× avg(10min) за последние 5 дней → momentum в направлении первых 30 мин
2. **Lunch Reversal (13:00-14:00)**: Перед дневным клирингом (14:00) часто происходят развороты. MM фиксирует позиции. Сигнал: цена выросла с 10:00 до 13:00 → SHORT перед 14:00
3. **Close Sweep (18:00-18:45)**: Последние 45 мин основной сессии. Если объём > 3× нормы и цена у уровней → Stop hunt перед вечерней сессией

**Реализуемость**: 9/10 — просто, данные есть, тест на истории за 1-2 дня

### Идея 7: Intraday OI Spike (tradestats_fo — 5m OI)

**Источник данных**: `tradestats_fo` (oi_close на 5m барах)

**Сигнал**: OI на MOEX futures обновляется в реальном времени (в отличие от US рынков). Резкий скачок OI за 5-10 минут — аномалия.

**Логика**:
1. oi_spike = oi_close / oi_close.shift(1) — изменение OI за 5 минут
2. z_oi_spike = z-score(log(oi_spike), 30 баров)
3. Сигнал: |z_oi_spike| > 3.0 → экстремум
4. Если OI↑↑ (z > 3) и цена↑: новые длинные позиции открываются → продолжаем вверх
5. Если OI↑↑ (z > 3) и цена↓: новые короткие позиции → продолжаем вниз
6. Если OI↓↓ (z < -3) и цена↓: короткие закрываются → разворот вверх (short squeeze)
7. Если OI↓↓ (z < -3) и цена↑: длинные закрываются → разворот вниз (long liquidation)

**Реализуемость**: 6/10 — 5m OI есть в tradestats_fo, но нужно проверить, real-time ли обновляется OI (на MOEX — да, но в CH может быть close-of-bar)

### Идея 8: Cross-Ticker Correlation Surge (tradestats_fo + prices_5m_oi)

**Источник данных**: Все 238 tickers в tradestats_fo

**Сигнал**: MOEX имеет структурные пары: Si/EU (USD/RUB + EUR/RUB), BR/CR (Brent + Crude), GZ/AF (газ, алюминий). Когда корреляция между парами резко меняется — это сигнал макро-события.

**Логика**:
1. rolling_corr(Si_ret, EU_ret, 20 баров) — корреляция USD/RUB и EUR/RUB
2. Si_ret = z-score(close_Si.pct_change(), 5m)
3. Когда корреляция Si/EU падает с >0.8 до <0.3 за 1 час → структурный сдвиг
4. Если EU слабее Si → LONG EU/Si pair
5. Дополнительно: BR и CR (нефть Brent и Urals). Корреляция обычно >0.9. Падение корреляции → арбитражный сигнал.

**Реализуемость**: 7/10 — данных много, требует cross-ticker запросов к CH

### Идея 9: OI Flat + Volume Explosion (tradestats_fo)

**Источник данных**: `tradestats_fo` (oi_close, vol_sum)

**Сигнал**: Классическая дивергенция: OI не меняется (flat), но объём резко растёт (vol_z > 2). Это означает, что старые позиции переходят из рук в руки (churn), а не новые открываются. После churn часто следует разворот.

**Логика**:
1. oi_flat = |oi_close - oi_close.shift(5) / oi_close.shift(5)| < 0.01 (OI почти не изменился за 5 баров)
2. vol_surge = vol_sum / vol_sum.rolling(20).mean() > 2.0
3. Сигнал: oi_flat AND vol_surge → churn → разворот
4. Если до churn тренд был вверх → SHORT, если вниз → LONG

**Реализуемость**: 9/10 — самые базовые данные, нет сложных вычислений

### Идея 10: Futures-Forward Roll Over Signal

**Источник данных**: Если есть данные о контрактах (с разными датами экспирации) — `securities` (содержит ГО, lot, stepprice, minstep)

**Сигнал**: Когда объём и OI начинают перетекать из ближнего контракта в дальний — это сигнал о росте интереса к следующему месяцу. Раннее обнаружение нового тренда.

**Логика**:
1. Для каждого ticker есть несколько контрактов (Si-6.26, Si-9.26, Si-12.26 и т.д.)
2. Рассчитать долю OI ближнего контракта в общем OI по всем контрактам
3. Если доля падает ниже 50% → roll is in progress
4. Если roll начался раньше обычного (за 10+ дней до экспирации) → сильный интерес к следующему месяцу → сигнал направления

**Реализуемость**: 5/10 — зависит от наличия данных о контрактах в CH. Нужно проверить securities table и наличие multiple contracts per ticker.

## 4. Протокол тестирования для каждой идеи

### Общие правила:
- **NO look-ahead bias**: rolling z-score, shift(-1) для forward return
- **Комиссии**: 4 руб/контракт × 2 (round-trip) = 8 руб
- **ГО**: из таблицы securities (или фиксированные значения)
- **Минимальное число сделок**: 30 для статистической значимости
- **Walk-forward**: sequential по месяцам (не random)

### По каждой идее:

| Идея | Данные | Запрос CH | Период | Время теста |
|------|--------|-----------|--------|-------------|
| 1. Stop Hunt | alerts_fo + tradestats_fo | `SELECT * FROM moex.alerts_fo WHERE ...` + 5m OHLC | 2024-2026 | 2 дня |
| 2. HHI divergence | hi2_fo + tradestats_fo | `SELECT * FROM moex.hi2_fo WHERE ticker=X` | 2024-2026 | 1 день |
| 3. FIZ/YUR divergence | prices_5m_oi | `SELECT * FROM moex.prices_5m_oi WHERE symbol=X` | 2025-2026 | 2 дня |
| 4. Volume Profile | tradestats_fo | `SELECT time, close, vol_sum FROM moex.tradestats_fo` | 2024-2026 | 1 день |
| 5. Disb_z + OI_z | tradestats_fo | `SELECT disb, oi_close, close FROM ...` | 2024-2026 | 1 день |
| 6. Session Patterns | tradestats_fo | `SELECT * FROM moex.tradestats_fo` с фильтром по времени | 2024-2026 | 1 день |
| 7. OI Spike 5m | tradestats_fo (oi_close) | `SELECT oi_close FROM ... ORDER BY time` | 2024-2026 | 1 день |
| 8. Cross-ticker corr | tradestats_fo | Множественные запросы Si, EU, BR, CR | 2024-2026 | 2 дня |
| 9. OI Flat + Vol Expl | tradestats_fo | `SELECT oi_close, vol_sum FROM ...` | 2024-2026 | 1 день |
| 10. Futures Roll | securities + tradestats | `SELECT * FROM moex.securities` + query by contract | 2024-2026 | 2 дня |

### Пример скрипта проверки (Python-псевдокод):

```python
# Для Stop Hunt Detection
import pandas as pd
import numpy as np
from clickhouse_driver import Client

ch = Client('10.0.0.60')

# 1. Загрузить 5m бары для Si (пример)
bars = ch.query_df("""
    SELECT time, ticker, pr_open, pr_high, pr_low, pr_close, vol_b, vol_s
    FROM moex.tradestats_fo
    WHERE ticker = 'Si' AND time >= '2024-01-01'
    ORDER BY time
""")

# 2. Найти потенциальные stop hunt
lookback = 20
bars['max_prev'] = bars['pr_high'].rolling(lookback).max().shift(1)
bars['min_prev'] = bars['pr_low'].rolling(lookback).min().shift(1)

# False breakout upward
bars['fake_high'] = (bars['pr_high'] > bars['max_prev']) & \
                    (bars['pr_close'] < bars['pr_high'] - 0.3 * (bars['pr_high'] - bars['pr_low']))

# False breakout downward
bars['fake_low'] = (bars['pr_low'] < bars['min_prev']) & \
                   (bars['pr_close'] > bars['pr_low'] + 0.3 * (bars['pr_high'] - bars['pr_low']))

# 3. Forward return
bars['ret_fwd'] = bars['pr_close'].shift(-1) / bars['pr_close'] - 1

# 4. Результаты
signals = bars[bars['fake_high'] | bars['fake_low']]
print(f"Total signals: {len(signals)}")
print(f"Win rate: {(signals['ret_fwd'] > 0).mean():.1%}")
print(f"Avg return: {signals['ret_fwd'].mean():.4f}")
```

## Ключевые инсайты TRIZ

1. **Принцип «недостаточности действия»**: CVD не работает, потому что vol_b ≈ vol_s всегда. Нужно переключиться на данные, где есть **реальная асимметрия информации** — fiz/yur, HHI, alerts.

2. **Принцип «локальности»**: На MOEX edge нужно искать на коротких таймфреймах (5-30m), а не на дневных. Комиссия съедает дневную доходность.

3. **Принцип «отбрасывания шума»**: 5 из 10 идей — это события (event-based), а не непрерывные индикаторы. События имеют более высокий SNR, чем осцилляторы.

4. **Принцип «неиспользованного ресурса»**: alerts_fo (331K записей) и hi2_fo (1.14M записей) — данные, которые никто не анализирует. В них потенциально наибольший edge.
